# Codebase Concerns

**Analysis Date:** 2026-03-02

## Incomplete Authentication Implementation

**Auth Flow Not Implemented:**
- Issue: Sign-in, sign-up, forgot-password, and reset-password forms contain placeholder TODOs; form submissions display toast notifications instead of executing actual auth logic
- Files:
  - `apps/web/app/(auth)/sign-in/sign-in-form.tsx` (lines 48-49)
  - `apps/web/app/(auth)/sign-up/sign-up-form.tsx` (lines 34-35, 42-43)
  - `apps/web/app/(auth)/forgot-password/forgot-password-form.tsx` (lines 35-36)
  - `apps/web/app/(auth)/reset-password/reset-password-form.tsx` (lines 34-35)
- Impact: OAuth sign-in/sign-up cannot be triggered; password reset emails cannot be sent; account creation and password resets are non-functional
- Fix approach:
  1. Implement OAuth handlers using `useAuth().signInWithOAuth(provider)` in sign-in and sign-up forms
  2. Implement forgot-password to call `supabase.auth.resetPasswordForEmail(email)` and navigate to reset page
  3. Implement reset-password to call `supabase.auth.updateUser({ password })` after validating reset token
  4. Implement sign-up to call `useAuth().signUp({ email, password })` after validating email is not already registered
  5. Ensure actual sign-in works by validating against test auth credentials from `.env`

## Missing Feature Router Procedures

**Users Router Uses Stub Data:**
- Issue: `usersORPCRouter` handlers return hardcoded mock data (John Doe, Jane Smith, random IDs) instead of querying Supabase
- Files: `packages/features/users/src/routers/usersORPCRouter.ts` (lines 7-50)
- Impact: User listing, retrieval, and creation do not persist; no database integration; all data is ephemeral and loses context across requests
- Fix approach:
  1. Replace `list()` handler to query `context.supabase.from("users").select("*")`
  2. Replace `get()` handler to query `context.supabase.from("users").eq("id", input.id).single()`
  3. Replace `create()` handler to insert into `context.supabase.from("users").insert({ name, email })`
  4. Ensure all queries respect Row-Level Security (RLS) via authenticated Supabase clients in context
  5. Add error handling for missing users (return null from get) and duplicate emails (throw on create)

## Generated Router Types Require Manual Regeneration

**Type Sync Risk:**
- Issue: `packages/infrastructure/api-client/src/generated/router-types.d.ts` is generated by `pnpm gencode` and must be manually regenerated after any change to `apps/api/src/router.ts` or feature routers; the generator validates that feature types don't leak into the output
- Files:
  - Generated: `packages/infrastructure/api-client/src/generated/router-types.d.ts`
  - Generator: `apps/api/scripts/generate-router-types.ts`
  - Source: `apps/api/src/router.ts`
- Impact: Consuming apps (web, mobile) will have stale Router types if `pnpm gencode` is not run after router changes; type mismatches cause runtime errors; if feature types leak, the script deliberately fails with validation error
- Fix approach:
  1. Commit reminder in CLAUDE.md: run `pnpm gencode` after editing `apps/api/src/router.ts` or feature routers
  2. Add pre-commit hook to detect router changes and enforce `pnpm gencode` (check git diff `apps/api/src/router.ts` and regenerated file)
  3. Run `pnpm gencode` in CI to detect missing regenerations on pull requests
  4. Validate generator logic: ensure feature types are fully expanded before writing (currently checks for `@features/` in output and throws if found)

## Metro Config Does Not Exclude Unsafe Resolver

**Potential UniWind Breakage:**
- Issue: `apps/mobile/metro.config.js` does not explicitly disable `config.resolver.unstable_conditionNames`; if any dependency sets this override, it breaks UniWind's platform-aware resolution for web target (causes `createOrderedCSSStyleSheet` resolution failures)
- Files: `apps/mobile/metro.config.js` (lines 1-19)
- Impact: Mobile web export may fail silently; CSS utilities from UniWind are not found; conditional exports for web target are ignored
- Fix approach:
  1. Add explicit comment and validation: ensure `config.resolver.unstable_conditionNames` is never set
  2. If any dependency requires it in future, use a different resolution strategy or pin that dependency version
  3. Document in CLAUDE.md: "Never set `unstable_conditionNames` in metro.config.js — it breaks UniWind's web resolver"

## Mobile Theme Tokens Must Stay Synchronized

**Hard-Coded HSL Values in Two Locations:**
- Issue: `apps/mobile/global.css` contains hard-coded theme color tokens (HSL values) that must match `packages/infrastructure/ui/src/globals.css` `:root` and `.dark` blocks; UniWind requires both `@variant light` and `@variant dark` blocks to function — missing either causes always-dark rendering
- Files:
  - Web source: `packages/infrastructure/ui/src/globals.css` (`:root` and `.dark` blocks)
  - Mobile target: `apps/mobile/global.css` (lines 14-60, theme variants)
- Impact: Updating web theme colors requires manual sync to mobile; mismatched colors cause visual inconsistencies; missing `@variant light` block causes always-dark UI on mobile
- Fix approach:
  1. Add validation script: parse both files and verify HSL values match (warn if they don't)
  2. Document manual sync requirement in CLAUDE.md with a checklist
  3. Consider generating mobile theme from web globals.css via build script (future improvement)
  4. Update pre-commit hook to validate theme sync (compare color tokens in both files)

## Tailwind v4 Scanning Requires Explicit @source

**Risk of Uncompiled Component Styles:**
- Issue: Tailwind v4 is CSS-first and does not auto-scan imported packages; each consuming web app must add `@source "../node_modules/@infrastructure/ui-web/src";` in its `app/globals.css` to include component styles; if this is omitted, components render without styling
- Files:
  - `apps/web/app/globals.css` (line 3)
  - `apps/landing/app/globals.css` (line 3)
- Impact: New web apps (future sibling to web/landing) won't compile component styles unless developer remembers to add `@source` directive; unstyled components break UI
- Fix approach:
  1. Create template for new web apps with `@source` directive pre-added
  2. Document in CLAUDE.md under "New web app checklist": "Add `@source ../node_modules/@infrastructure/ui-web/src;` to `app/globals.css`"
  3. Consider adding a shared CSS file in `@infrastructure/ui-web` that re-exports the `@source` directive (if Tailwind v4 allows import delegation)

**Form Boilerplate Not Abstracted:**

- Files:
  - `apps/web/app/(auth)/sign-in/sign-in-form.tsx`
  - `apps/web/app/(auth)/sign-up/sign-up-form.tsx`
  - `apps/web/app/(auth)/forgot-password/forgot-password-form.tsx`
  - `apps/web/app/(auth)/reset-password/reset-password-form.tsx`
- Impact: Bug fixes to form styling or validation display must be applied 4 times; future auth forms require copying boilerplate; high maintenance burden
- Fix approach:
  1. Extract a generic `AuthFormTemplate` component that accepts `schema`, `title`, `description`, `fields`, and `onSubmit`
  2. Use a schema-driven approach: define fields from Zod schema metadata (labels, types, placeholders)
  3. Replace each form with a thin wrapper that imports the template and provides only custom logic
  4. Ensure OAuth buttons are extracted to a reusable `OAuthButtons` component

## Missing Environment Validation

**Runtime Errors from Missing Env Vars:**
- Issue: `apps/api/src/app.ts` manually checks `SUPABASE_PUBLISHABLE_KEY` (line 14) but silently falls back for `SUPABASE_URL`; consuming apps don't validate required env vars on startup; missing env causes unclear runtime failures later
- Files:
  - `apps/api/src/app.ts` (lines 7-14)
  - `apps/api/src/index.ts` (line 7)
  - `apps/web/lib/navigation.tsx` (assumes env is present)
  - `apps/mobile/app/index.tsx` (assumes env is present)
- Impact: Developers forget to copy `.env.example` → `.env`; app starts but crashes on first API call or auth attempt; debugging is opaque
- Fix approach:
  1. Create a shared `validateEnv()` function in `@infrastructure/utils` that checks required vars (Supabase URL, publishable key, API URL for web/mobile)
  2. Call it in app entry points before rendering: `apps/api/src/index.ts`, `apps/web/app/layout.tsx`, `apps/mobile/app/index.tsx`
  3. Throw clear error with instructions: "Missing NEXT_PUBLIC_SUPABASE_URL — copy `.env.local.example` to `.env.local` and fill in values from `pnpm --filter supabase-db start`"
  4. Test locally by deleting env vars and verifying error message appears

## No Rate Limiting or Throttling on API

**Brute Force Attack Vector:**
- Issue: Hono app has no rate limiting middleware; sign-in attempts are unlimited; password reset emails can be spammed; no CORS rate limit headers
- Files: `apps/api/src/app.ts` (lines 1-67)
- Impact: Attackers can brute-force passwords; email spam; API exhaustion; no protection against automated attacks
- Fix approach:
  1. Add rate limiting middleware (e.g., `hono/rate-limiter` or custom Redis-based counter)
  2. Rate limit by IP and by endpoint:
     - Sign-in: 5 attempts per 15 minutes
     - Password reset email: 3 attempts per hour per email
     - General API: 100 requests per minute per IP
  3. Return 429 (Too Many Requests) with Retry-After header
  4. Log rate limit violations for security monitoring
  5. Document in CLAUDE.md: rate limits are in place, mention bypass for local dev if needed

## CORS Hardcoded to Localhost

**Deployment Configuration Risk:**
- Issue: CORS origins are hardcoded to `["http://localhost:3000", "http://localhost:3001"]` with fallback from `CORS_ALLOWED_ORIGINS` env var; if env var is missing, defaults to localhost only (production would fail)
- Files: `apps/api/src/app.ts` (lines 21-24)
- Impact: Production deployment will reject requests from real domains unless `CORS_ALLOWED_ORIGINS` is explicitly set; misconfigurations could allow all origins or block legitimate clients
- Fix approach:
  1. Make `CORS_ALLOWED_ORIGINS` required (throw if missing in production)
  2. Validate origins format in startup: ensure each origin is a valid URL
  3. Document in CLAUDE.md: "Set `CORS_ALLOWED_ORIGINS` before deploying (comma-separated, no wildcards)"
  4. Consider: allow `*` only if explicitly opted in via env flag (e.g., `ALLOW_ANY_ORIGIN=true`)

## No Input Sanitization Before Database

**SQL Injection or RLS Bypass Risk:**
- Issue: User routers accept `CreateUserSchema` and `GetUserSchema` inputs but don't validate against malicious patterns (e.g., script payloads in email field); Supabase RLS should protect, but no explicit input cleansing
- Files: `packages/features/users/src/routers/usersORPCRouter.ts` (lines 25-49)
- Impact: Malicious input could be stored in database; while RLS prevents unauthorized reads, stored XSS or LDAP injection could affect application logic
- Fix approach:
  1. Validate email format strictly: use `z.email().max(100)` in schemas (already done in auth schemas, ensure consistency)
  2. Sanitize string inputs: remove control characters, enforce max lengths
  3. Add DOMPurify or similar for any user-generated content that displays in UI
  4. Test: attempt to insert `<script>alert('xss')</script>` in name field, verify it's stored as-is and escaped in UI

## No Error Boundaries in React Components

**Unhandled Component Crashes:**
- Issue: Web app components don't use Error Boundaries; if a component throws, React unmounts the entire subtree with no fallback UI
- Files: `apps/web/app/layout.tsx`, `apps/landing/app/layout.tsx`
- Impact: Any unhandled component error crashes the entire page; users see blank white screen; no recovery path
- Fix approach:
  1. Add Error Boundary wrapper in root layout (or use Next.js 13+ error.tsx)
  2. Catch and display error message: "Something went wrong. Please refresh the page."
  3. Log error to monitoring service (Sentry, LogRocket, etc.)
  4. Test: throw error in a nested component, verify boundary catches it
