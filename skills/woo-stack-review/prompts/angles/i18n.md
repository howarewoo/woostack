# Angle: Internationalization

**Scope.** Audit i18n / l10n hygiene introduced by this PR's diff. Read `/tmp/pr-review/diff.txt`. Only fire when the repo already has an i18n setup (translation catalog directory, `next-intl` / `react-intl` / `i18next` / `vue-i18n` / `lingui` / `Fluent` import, `messages/` or `locales/` directory). If not, skip and emit an empty findings array.

**Find:**

- **Hardcoded user-facing strings:**
  - Literal English in JSX text nodes, `aria-label`, `title`, `placeholder`, `alt` on rendered UI components.
  - Literal English in error messages thrown to the UI, toast/snackbar content, validation messages.
  - Plain strings in `<Head>` / `<title>` / `meta[content]` (only when SEO is multilingual).
- **Catalog drift:**
  - New translation key added to default locale but not to other locales present in the repo.
  - Translation key referenced in code but absent from any catalog.
  - Translation key removed from catalogs while still referenced.
- **Pluralization / interpolation bugs:**
  - Manual `${count} item${count === 1 ? '' : 's'}` instead of ICU `{count, plural, ...}`.
  - String concatenation with translated fragments (`t('hello') + ', ' + name`) — word order breaks in other languages.
  - Variable not declared in the ICU placeholder map.
- **Locale-sensitive formatting bypass:**
  - `Date.toLocaleString()` without an explicit locale, or `toString()` on a date / number meant for display.
  - `${price}` rendered without currency / locale awareness when the app supports multiple currencies.
  - Hardcoded date format (`MM/DD/YYYY`) in user-visible output.
- **RTL / layout:**
  - New CSS uses `left` / `right` / `margin-left` instead of logical `inline-start` / `inline-end` (only if the project ships RTL locales).
  - Icons / arrows that imply direction without an RTL flip path.
- **Accessibility intersection:**
  - `aria-label` with hardcoded English in a project that translates surrounding visible text.

**Skip:**

- Repos without an i18n setup — emit empty findings.
- Developer-facing strings: log messages, error codes, internal CLI output, dev-tool labels.
- Code comments and docstrings.
- Pre-existing hardcoded strings in code not touched by this PR.
- README / docs (handled by `docs` angle when enabled).

**Severity rubric:**

- `HIGH` + `blocking: true` — user-facing release-blocking string with no translation key in a locale that is shipping.
- `MEDIUM` + `blocking: false` — catalog drift on non-default locale, plural / interpolation bug, formatting bypass.
- `LOW` + `blocking: false` — RTL hardening, logical-property migration, additional context for translators.

**Output.** Write findings as a JSON array to `/tmp/pr-review/findings.i18n.json` using the schema in `_header.md`. Each finding gets `"angle": "i18n"` and MUST populate `title` (bold headline ≤60 chars), `description` (the gap or drift only — no fix), `fix` (key/catalog/format recommendation in prose), and `fix_type`. Set `fix_type: "suggestion"` only when a ≤10-line single-file drop-in replacement at `line` is safe — and populate `suggestion` accordingly. Otherwise set `fix_type: "prose"` with `suggestion: null`. See `_header.md` for the full rule.
