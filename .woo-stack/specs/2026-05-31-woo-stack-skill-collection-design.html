woo-stack — Skill Collection Redesign

Status: Design / pending approval Date: 2026-05-31 Branch: pivot-to-skill

1. What woo-stack becomes

woo-stack stops being a single bootstrap skill and becomes an opinionated collection of orchestration skills. Each skill is thin glue that sequences existing, proven skills in a fixed order, inherits their gates, and — on completion — offers the natural next step. The value is the sequence and the handoffs, not re-implementing brainstorming, planning, or review.

Packaging stays a multi-skill skills collection (the npx skills add path), not a Claude Code plugin. skills-lock.json already keys multiple skills per source repo by skillPath, so one repo publishing several skills is native.

2. Skill inventory

woo-stack-review and woo-stack-address-comments are two command entry points over one shared ported woo-review engine (scripts + prompts).

Command relationship

Discrete commands — each invoked on its own, each ending by offering the next step. The user stays in control and never has to remember the sequence. Nothing chains automatically; nothing merges.

build→ offer PR → review→ offer → address-comments→ offer re-review → review→ … → user merges (outside woo-stack)

3. woo-stack-build in detail

The core deliverable. Thin glue: it sequences the sub-skills and inherits their gates only (brainstorming requires design approval; writing-plans and executing-plans have their own checkpoints). It adds no extra phase gates of its own.

brainstorming→ write spec (HTML)→ grill-me→ writing-plans→ executing-plans→ ask: open PR?

PR-sized decomposition

build steers work toward well-scoped PRs of preferably ≤500 lines of code. This is a soft target that shapes planning, not an enforced gate. When the spec implies more change than fits one reviewable PR, build:

  • decomposes during planning — writing-plans structures the work as a sequence of independently shippable increments, each sized to land as one small PR;
  • runs one increment per build cycle — each cycle ends by offering a PR for that slice, so a large project becomes a chain of small reviewable PRs rather than one giant diff;
  • flags oversize slices — if an increment can't reasonably stay under the target, build says so and proposes a further split before executing.

The 500-LOC figure is a guideline for reviewability; genuinely atomic changes that exceed it are allowed when splitting would be artificial.

Output overrides (diverges from superpowers defaults)

superpowers writes markdown specs to docs/superpowers/specs/. build overrides both location and format, so its SKILL.md explicitly directs the write step rather than inheriting it:

ArtifactFormatLocation
SpecSelf-contained styled HTML (inline CSS, diagrams) — no markdown source kept.woo-stack/specs/
PlanMarkdown (working checklist).woo-stack/plans/

HTML specs are for visualization — richer than plain markdown. grill-me hardens the HTML spec in place. Plans remain markdown because they are working checklists, not visualization artifacts. This very document is the first instance of the HTML-spec convention.

4. Dependency model

Porting woo-review in shrinks external skill dependencies to superpowers (brainstorming, writing-plans, executing-plans) and grill-me — used by build alone. woo-review is internalized: it carries its own binary requirements (gh, jq, node) and its optional review-angle skills as recommends.

Preflight (shared behavior, primarily build)

  • Declared at install — the install script / lock pulls each required skill (npx skills add per dep). Most users never hit a gap.
  • Checked at skill start — each orchestrator preflights its own deps.
  • Missing → offer inline install and continue — name exactly what's missing; do not end the task and force a re-invoke.
  • Declined → methodology deps (superpowers, grill-me) fall back to following the principle manually, stated clearly as degraded. (woo-review no longer applies — it's first-party.)

5. review + address-comments (ported woo-review)

woo-review is ported into this repo in full and the upstream howarewoo/woo-review repo is deprecated (deprecation notice lives in that repo — follow-up, out of scope here).

  • review — runs the parallel review swarm, validates findings with the Skeptical Validator, posts a single batched GitHub review when a PR is targeted (or reviews the local diff with no posting).
  • address-comments — drives woo-review's existing address verb: fetch unresolved threads, restate, verify against the code, choose FIX / ACCEPT (push back) / CLARIFY, reply without performative language, resolve, push. The receiving-code-review judgment is already embedded — no superpowers dependency.

Unified consumer working directory: .woo-stack/

When ported, woo-review's per-repo state directory is renamed from .woo-review/ to .woo-stack/ in the consumer repo, so the whole collection shares one working dir. Every .woo-review/... reference across woo-review's scripts, SKILL.md, and prompts is rewritten:

.woo-stack/metrics.json is gitignored by default. It is a local-only rolling aggregate, not shared team state. The skill ensures the ignore entry exists (adds it to .gitignore when first writing metrics). The other .woo-stack/ files stay tracked: config.json and memory.md are shared team state; specs/ and plans/ are artifacts.
This is the consumer-repo state dir rename only. It is distinct from the internal skills/woo-review/scripts/... source paths, which change because the skill directory itself moves (see Repo changes).

6. Repo changes

  • skills/woo-stack/ → skills/woo-stack-bootstrap/ (SKILL.md + references/ moved intact).
  • New skill dirs: skills/woo-stack-build/, skills/woo-stack-review/, skills/woo-stack-address-comments/, plus the shared ported woo-review engine.
  • Promote woo-review from .agents/skills/woo-review/ (dev copy) to first-party skills/ (canonical). The repo then dogfoods its own review skill.
  • Rewrite the consumer state dir .woo-review/ → .woo-stack/ everywhere in the ported woo-review (scripts, SKILL.md, prompts) — see section 5.
  • references/development.md → trimmed to a pointer (the loop is automated by the orchestrator skills) plus the branching model, which stays prose-worthy. The skills become source of truth for the loop; no duplicated step lists.
  • README.md / AGENTS.md → document the four commands; note the woo-review repo is deprecated and lives here now; update layout + quick-reference tables + the two-modes framing.
  • skills-lock.json → drop the woo-review entry (now first-party); keep superpowers + grill-me.

7. Naming

Commands are hyphenated: /woo-stack-bootstrap, /woo-stack-build, /woo-stack-review, /woo-stack-address-comments. The literal colon form (woo-stack:build) is a Claude Code plugin feature; the skills-collection packaging chosen here uses flat skill names, so the colon is not available without restructuring as a plugin.

8. Non-goals

  • No merge automation — woo-stack stops short of merge; the user merges through their own process / branch protection.
  • No plugin restructure — stays a skills collection.
  • No application code in this repo (markdown + skill assets only).
  • Bootstrap behavior is unchanged — renamed and repackaged, not redesigned.
  • Actually deprecating the old woo-review repo is a follow-up in that repo, not part of this work.
